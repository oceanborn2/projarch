SourceForge.net Data Backup and Restoration Policy

This document provides an overview of the backup and restore procedures followed by SourceForge.net staff, as well as instructions for both SourceForge.net users and projects hosted on SourceForge.net which may be used in backing up their project data. It is important to recognize that all SourceForge.net users and projects are responsible for their own content. This guide provides an easy-to-follow checklist of data that projects should backup themselves. Backups are only good if they are generated BEFORE a problem occurs; take this seriously now in case you need your backup later.


What data is stored on SourceForge.net?

SourceForge.net provides a number of services; these services depend on a large amount of disk capacity for end-user and site data storage. SourceForge.net-hosted data falls in to five basic categories:
  • SourceForge.net site data consists of all data placed on the SourceForge.net site, including user and project records, forum postings, site configuration details, and Tracker (bug reports, support requests, etc.) postings.
  • SourceForge.net host data consists of system files and core operating system data for all hosts serving data for SourceForge.net. i.e. project web servers, project shell servers, project CVS servers, etc.
  • Project file release data consists of all files released through the SourceForge.net file release system, which are then placed on the network of high-capacity download servers.
  • Project CVS data consists of all materials which have been imported or added to project CVS repositories for projects hosted on SourceForge.net.
  • Project-specific and user-specific content consists of materials placed on the compile farm hosts, in user home directories, and in the project web space for projects.


What data is backed up by SourceForge.net?

As stated, SourceForge.net classifies data in to five categories. All data hosted on SourceForge.net is backed up by SourceForge.net at regular intervals. These backups are full backups (a type of backup), rather than incremental or differential backups. Based on the amount of hosted data and the type of backup we use, backups take a fairly long amount of time to occur (many hours) each day.


How frequently do backups occur?

Backups which involve user, project and SourceForge.net site data occur daily, generally occurring during the morning hours (PST/PDT time zone) each day. Exact backup times are not guaranteed; backups may be delayed as result of SourceForge.net site outages or planned maintenance windows. System data backups occur generally occur at daily intervals, though this schedule may change depending on current maintenance events and planned upgrade events.


What is the purpose of SourceForge.net data backup?

SourceForge.net backups are used to provide restoration of hosts and data in the event of catastrophic, site wide issues. These backups are also used to ensure suitable response time may be guaranteed in the event that disaster recovery (i.e. catastrophic loss of all hosts as result of earthquake, fire, etc.) should be required. SourceForge.net backups are not used to provide user-requested file restoration; based on the resource limitations of SourceForge.net, this is not a service we provide. SourceForge.net backups are used solely for resolution of significant, site wide issues, such as host hardware failure or catastrophic disaster. Each project is responsible for backing up their own data.


As a SourceForge.net project, what data should we back up?

Each project hosted on SourceForge.net is responsible for their own content and data backups. Please ensure that your project has addressed the following backup concerns:
  • Remind your developers to backup their data.
  • Backup your file release data.
  • Backup your project web space.
  • Backup the contents of your MySQL database.
  • Backup your CVS repository content regularly.
  • Ensure that key members of your team are aware of your backup policy and procedures.
  • Ensure that mailing list admin passwords and other project-specific security content has been retained in a secure location and is accessible to key members of your project team.
  • Important: If your project stores files within its project web space (which are flagged as world writable or are otherwise writable by the project web server) it is vital that you ensure proper backup of those files occurs on a regular basis. Files of this nature are prone to damage, by their very nature; please protect them appropriately, through the use of regular backups.


As a SourceForge.net user, what data should I back up?

As a SourceForge.net user, first and foremost, you should be aware of the backup policies and strategies used by your project. In the event that there is loss of project data, it is important that all key members of that project group know how to proceed.

Second, it is important that you backup the contents of your home directory on the project shell servers and backup any materials you have placed on the SourceForge.net compile farm. Finally, please ensure you have made an adequate backup of your SSH key data and any project-related password data you may have received.


How may a SourceForge.net project backup their file releases?

File releases placed on the SourceForge.net download server network, through use of the SourceForge.net file release system, automatically enjoy an enhanced level of data redundancy and synchronization. The SourceForge.net download server network periodically mirrors changes from a centralized host to a number of geographically diversified hosts, all with redundant storage hardware. This helps to ensure stability, both short-term and long-term, within the download server network, helping to ensure accessibility of your file releases to your end-user base.

As with all other project-provided data, it is important that you retain a set of local backups of your file releases. We recommend that you retain original file release archives at time of file release creation. We also recommend that, for data security purposes, you sign each of your releases with GNU Privacy Guard or PGP. Additional documentation regarding file release security may be found in our Guide to the File Release System.


How may a SourceForge.net project backup their project web space?

SourceForge.net recommends that all projects retain an off-site backup of all project group directory space content which has been placed on the project shell servers (and project web servers). Project group data may be copied off of the project shell server in two ways. It should be noted that all data you have uploaded to your project web space is accessible through the project shell server; you will use the project shell server, not the project web server, to backup this data.

The project group directory for your project is located in a directory hierarchy, split out by the first letter, and the first and second letter, of your project name.

Examples:

Project UNIX name: projectname
Directory structure: shell.sourceforge.net:/home/groups/p/pr/projectname

Project UNIX name: foobar
Directory structure: shell.sourceforge.net:/home/groups/f/fo/foobar

The first means to copy data off of the project shell server is to use scp, Secure CoPy, from your SSH suite. Specific instructions for using your SCP client may be found in the documentation accompanying your SSH client.

The second means to copy data off of the project shell server is to use rsync in conjunction with your SSH client (only available on some platforms). rsync permits you the means to reduce the bandwidth required to backup your project group data; this is particularly useful when you wish to make regular or frequent backups of this data. More information regarding rsync may be found on the Rsync project web site.

The SourceForge.net support team is happy to assist with any issues related to the services we offer, including file backup and restoration. To contact the SourceForge.net support team, first login to the SourceForge.net web site, then submit a support request.


How may a SourceForge.net project backup their MySQL data?

Each project hosted on SourceForge.net is eligible to use the project MySQL database services provided by SourceForge.net. Each project is responsible for their own data. As access to the project database servers is restricted, backups should be generated in one of two ways.

If your project is using its MySQL database resources for hosting an application and that application provides specific instructions for data backup, please follow those instructions and procedures. Please ensure that any backups you create are treated in a secure manner and are offloaded from the project shell servers or project web servers.

A complete backup if your MySQL database may be created from the project shell servers by making use of the 'mysqldump' utility. Instructions for performing MySQL database backup are included in the MySQL documentation.

It should also be noted that a simple-to-use and simple-to-configure means of generating database content and schema backups exists in the phpMyAdmin suite. Please refer to the application documentation, MySQL documentation or phpMyAdmin documentation as appropriate.


How may a SourceForge.net project obtain a complete copy of their project CVS repository for backup purposes?

The most important data to most projects is the content they have placed in their project CVS repository; the most recent development materials for that project and the historical trail of project development. SourceForge.net makes backing up the entirety of your project CVS repository easy; a new tarball of your project CVS repository is generated daily and is made available for you to download. A link to your project's nightly CVS tarball may be found on the Project Admin page for your project.

It should be noted that these tarballs contain a complete copy of your project CVS repository, including control files and all historical file data, not just the latest versions of each file (this is how the nightly repository tarball differs from CVS snapshots provided by some projects). To maintain an off-site backup of your CVS repository, simply download the nightly CVS repository tarball for your project.

Nightly CVS repository tarballs are overwritten daily on the SourceForge.net servers; depending on your project needs, you may wish to retain (locally, not on SourceForge.net) tarballs generated from many dates, rather than just the most recent tarball. SourceForge.net retains only the latest nightly CVS tarball; this permits us to generally restore project CVS repository contents to within 24 hours of issue.

In the event that significant damage to your project CVS repository has occurred, please obtain the latest nightly CVS tarball for your project immediately, if you have not already done so. If you have a functional backup of your CVS repository, the SourceForge.net team will be able to recover your CVS repository; simply submit a support request. Once again, at first sign of an issue, please make a local backup of the nightly CVS tarball for your project; this will increase the chances that we will be able to recover your repository content.


How may a user backup their materials from the project shell servers?

User home directories on the project shell servers should be treated as temporary working space. Please ensure that you create and retain backups of this data based on the importance of the data you place in your home directory. Data may be copied from the project shell server to your remote workstation using rsync or scp, as discussed previously in this document. Please refer to the instructions accompanying your rsync or scp client for further assistance, or seek assistance from the SourceForge.net support team as necessary.


How may a user backup their materials from the compile farm?

All materials placed on the SourceForge.net compile farm should be backed up regularly. In order to backup data from the compile farm, please use 'scp' (Secure CoPy) on the project shell servers (shell.sourceforge.net) to copy your data from the compile farm (cf.sourceforge.net) to a subdirectory off of your project group directory (/home/groups/p/pr/projectname). Depending on the data you wish to backup, you may wish to generate a compressed tarball (.tar.gz) of that data before copying it to the project shell server, as to ease the complexity of your backup operation. Once data has been copied to the project shell server, follow the instructions provided for backup of group directory data to offload that backup from the project shell server.


At what point will SourceForge.net restore data from backup?

SourceForge.net staff will perform data restoration in the event that data is damaged as result of host failure (i.e. all storage within both primary and backup hosts fails) or as result of complete, catastrophic loss of all site hardware. SourceForge.net staff performs data backups on a regular basis, tests the functionality of the backup system, and is knowledgeable of our backup solution so as to ensure the shortest possible restore time in the event of a significant site wide data loss.


Will SourceForge.net perform data restoration if I make a mistake?

Each SourceForge.net user and project is responsible for their own data. SourceForge.net backups are used solely for resolution of catastrophic site wide issues. SourceForge.net will not provide on-demand backups or restoration. Please ensure that your project has suitable backups of all data hosted on SourceForge.net (user home directories, project web space, CVS content and file releases).


High risk file storage: world-writable files in project web space

Some web-based applications require files to be writable by the user used to run the web server. As result, some projects elect to flag files as world writable or to have the user/group ownership of a file changed to match that of the web server. This places file data at a high risk of compromise; it is important that projects recognize that and take precautions to back up data regularly, as appropriate for their particular application. Files damaged or removed as result of exercise of these risks will not be restored by SourceForge.net staff. Please ensure that your project data is backed up in an appropriate fashion.


Under what circumstances will SourceForge.net perform data restoration?

SourceForge.net staff will perform data restoration only in support of efforts to recover service after host failure or catastrophic site wide events. SourceForge.net staff will not perform on-demand backup or restoration operations; due to resource limitations, this is not a service that SourceForge.net is able to perform.


What type of hardware and software is used for data backup?

SourceForge.net currently makes use of the AMANDA backup software, hosted on a Linux-based machine, in conjunction with an Overland AIT-2 tape array. The AIT-2 tape technology is capable of holding 50GB of compressed data or 100GB of uncompressed data per tape, on average.

Additional live data backups are performed using rsync, cpio and database-specific replication. This allows us to generate less durable, but more frequent backups for specific SourceForge.net site components. Built in server replication and data redundancy functions are used, as available.


Where is backup media stored?

Backup media used by the SourceForge.net backup hosts (tapes, CD-ROMs, etc.) are stored in no less than two locations. The most recent backups are stored on-site with the backup servers. When each backup pool is full, it is rotated off-site to a secure media storage facility. Backup media is treated with the same level of security as live server data.


How often does the SourceForge.net team test backup validity?

SourceForge.net makes use of the Open Source AMANDA backup software. This software includes built in backup testing functionality which is used by the SourceForge.net team to verify basic functionality of all backups performed. In addition to these automated mechanisms, archived backups are tested every six months on a rotating basis, to ensure disaster recovery capability.


Where can I obtain support related to file backup and restoration?

The SourceForge.net support team is happy to assist with any issues related to the services we offer, including file backup and restoration. To contact the SourceForge.net support team, first login to the SourceForge.net web site, then submit a support request.
